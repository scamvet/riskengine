## What it does

Scores a URL for the likelihood that it is malicious, using lexical features
only: no network calls, no enrichment, no clocks. That is what lets the same
artifact serve the online path and the offline degradation path required by
ADR-004.

- Feature spec: `url_lexical_v5` (34 columns)
- Model: XGBoost 200 x depth 8
- Calibration: Platt (sigmoid)
- Artifact: model.onnx (1408.0 KB ONNX), parity to native within 3.6e-07
- Track: product. Trains only on the CC0 corpus (LEGAL.md row C4), per ADR-007.

## What this evaluation does not establish

Read this before quoting the number above.

**The corpus is a compilation, and some of its signal is provenance.** It was
assembled from five upstream sources with different collection methods. Three
features have been removed for encoding which file a row came from rather than
anything about the address: `had_scheme`, `scheme_is_https`, and the per-TLD
one-hot columns. Each removal *lowered* the reported score. A provenance audit
(`scripts/audit_provenance.py`) trains on nested feature subsets and finds that
performance on features whose meaning is intrinsic to the URL - brand, keyword,
entropy, structure signals - is **0.229**, against 0.797 on the full set.
Neither number transfers to production unchanged.

**The features built for this product's actual market barely fire here.**
`brand_domain_mismatch` is active in 0.44% of rows and `.in` domains are 0.26%,
because the corpus is Western and global while the brand list is India-tuned.
Those features are individually sharp - `brand_domain_mismatch` carries a 1.94x
lift where it fires - and collectively invisible at corpus scale. In the market
this product serves, that ratio inverts.

**Some labels are wrong.** The lowest-scoring "phishing" rows include ordinary
article URLs from established sites. Achievable recall is therefore capped
below 1.0 for reasons unrelated to modelling.

**The consequence.** Per ADR-006, the ML layer is not the primary evidence for
a consumer phishing verdict. Deterministic facts and the reasoning layer above
it carry more than originally planned, and the product metric is the Chapter B
golden set of real Indian scam artifacts scored by the full stack, not this
number.

## Known false positives on legitimate Indian domains

The corpus contains almost no Indian sites - 0.26% of rows are `.in` - so the
model has never seen a legitimate Indian bank, and no aggregate metric measured
on a Western corpus can reveal what it does with one. The check below scores
every `official_domains` entry in the brand list, which are sites this project
itself asserts are real. **3 of 55 were flagged amber or red.**

| domain | band | probability |
|---|---|---|
| `https://icicibank.bank.in/` | amber | 0.7701 |
| `https://www.jio.com/` | amber | 0.7095 |
| `https://hdfcbank.bank.in/` | amber | 0.5858 |

Treat this as the honest ceiling on using the score alone. It is the concrete
form of the transfer problem described above, and the direct reason ADR-006
places deterministic facts and the reasoning layer above this model rather than
behind it.

## Intended use and limits

Intended: pre-verdict scoring inside ScamVet's layered stack, and offline
scoring when enrichment is unavailable.

Not intended: a standalone allow/deny gate, an authority on any specific site,
or use outside the layered verdict architecture. A red band is a risk
assessment, not an accusation.
